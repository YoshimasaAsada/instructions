# Refactor

既存コードのリファクタリング改善案を分析し、提案する。

- **目的:** コードの責務分離・重複排除・可読性改善を行う（外部から見た振る舞いは変えない）
- **入力:** 対象のファイル / ディレクトリ / 関数名 + リファクタリングの目的
- **出力:** 改善案の一覧 + 承認後に修正したコード

---

## 事前確認事項

1. **対象範囲** — ファイル / ディレクトリ / 関数名など
2. **リファクタリングの目的** — 以下から選択（複数可）
   - 可読性の向上
   - 責務分離（SRP）
   - 重複排除（DRY）
   - テスタビリティ向上
   - パフォーマンス改善
   - 特に指定なし（全般的に改善）

---

## 実行フロー

### Phase 1: 現状の分析

対象コードを読み込み、以下の観点で問題を洗い出す:

| 観点 | チェック内容 |
|------|------------|
| 責務 | 1つの関数/クラスが複数の責務を持っていないか |
| 重複 | 同一・類似のロジックが複数箇所にないか |
| 命名 | 変数・関数名が意図を正確に表しているか |
| 複雑度 | ネストが深すぎないか、条件分岐が多すぎないか |
| 結合度 | 不必要に他モジュールと密結合になっていないか |
| サイズ | 関数が長すぎないか（目安: 30行以上は分割を検討） |

---

### Phase 2: 改善案の提示

検出した問題ごとに以下を提示する:

```
### 改善 1: {タイトル}

**問題:** {現状の問題点}
**理由:** {なぜ修正すべきか}
**対象:** `path/to/file:関数名`

**修正前:**
（該当コード）

**修正後:**
（改善後のコード）
```

改善案は影響度・難易度の観点で優先順位をつけて提示する。

---

### Phase 3: 実施

ユーザーの承認を得た改善案のみを実施する。

実施時のルール:

- **外部から見た振る舞いを変えない** — リファクタリングは内部構造の改善であり、入出力の変更は含まない
- **Step ごとにコミット可能な状態を保つ** — 中間状態でビルドが壊れないようにする
- **テストがある場合は都度実行** — 既存テストが通ることを各ステップで確認

---

### Phase 4: 検証

```bash
# テスト実行（プロジェクトに応じて）
npm run test
# lint / type check
npm run lint
```

---

### Phase 5: 完了報告

- 実施した改善の一覧
- 変更したファイルのパス
- テスト結果
- 見送った改善案とその理由（あれば）
