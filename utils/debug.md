# Debug

エラーログや再現手順を入力に、原因の仮説立てと調査・修正を行う。

- **目的:** エラーの根本原因を特定し、修正案と再発防止策を提示する
- **入力:** エラーログ / スタックトレース / 再現手順 / 発生環境
- **出力:** 原因分析 + 修正案（before/after）+ 再発防止の提案

---

## 事前確認事項

以下の情報をユーザーから収集する（わかる範囲で）:

1. **エラーログ / スタックトレース** — エラーの全文
2. **再現手順** — どういう操作・条件で発生するか
3. **期待する動作** — 本来どうあるべきか
4. **発生環境** — 本番 / ステージング / ローカルなど
5. **直近の変更** — 最近デプロイした内容・マージしたPRなど

---

## 実行フロー

### Phase 1: エラーの分析

エラーログ / スタックトレースから以下を特定する:

- **エラーの種類** — 構文エラー / ランタイムエラー / 型エラー / タイムアウト / etc.
- **発生箇所** — ファイルパス・行番号・関数名
- **直接原因** — エラーメッセージが示している問題

---

### Phase 2: 仮説の立案

エラーの分析結果をもとに、原因の仮説を優先度順に列挙する:

```
## 仮説一覧

### 仮説 1（可能性: 高）
{仮説の内容}
**根拠:** {エラーログや関連コードから読み取れる根拠}
**調査方法:** {確認するファイル・コマンド・手順}

### 仮説 2（可能性: 中）
...
```

---

### Phase 3: 調査の実行

優先度の高い仮説から順に調査する。

各仮説に対して:

1. 関連コードを読み込む
2. 仮説が正しいかを検証する
3. 結果を報告する:
   - **原因確定** → Phase 4 へ
   - **棄却** → 次の仮説を調査

---

### Phase 4: 修正案の提示

原因が特定できたら、修正案を提示する:

```
## 原因

{根本原因の説明}

## 修正案

**対象:** `path/to/file:行番号`

**修正前:**
（問題のあるコード）

**修正後:**
（修正したコード）

## 再発防止

- {同種のバグを防ぐための提案（テスト追加・バリデーション追加など）}
```

修正の実施はユーザーの承認後に行う。

---

### Phase 5: 修正の検証

修正後に以下を確認する:

- エラーが再現しないこと
- 既存テストが通ること
- 関連するテストを追加（必要に応じて）
