# Feature Planning

新機能の要件から、要件定義書・設計書・タスクリストの3ドキュメントを生成する。

- **目的:** 機能開発に必要なドキュメント一式を構造化して作成する
- **入力:** 機能の要件・改修対象リポジトリ・参考リポジトリ（任意）
- **出力:** `{ブランチ名}.requirement.md` / `{ブランチ名}.spec.md` / `{ブランチ名}.task.md`

---

## 事前確認事項

ドキュメントを作成する前に、ユーザーに以下を確認する（未確定の場合）:

1. **ブランチ名** — ファイル名のプレフィックスになる（例: `feat/add-video-watch-flags`）
2. **改修対象リポジトリ** — どのシステムを改修するか
3. **参考・別システム** — 真似したい実装があるか、あればどのリポジトリか

---

## 実行フロー

### Phase 1: コード調査

ユーザーの要件をもとに、改修対象・参考システムを調査する。
可能であれば改修対象と参考システムを並列で調査する。

**改修対象システムの調査ポイント:**

- 関連するDBモデル定義（ORM使用時はテーブル名・カラム名のマッピングを必ず確認）
- 関連するサービス・ユースケースの現在の実装
- 関連するスキーマ型定義
- 関連するフロントエンドのhooks・コンポーネント
- 既存のテスト

**参考システムの調査ポイント（ある場合）:**

- 参考にするユースケース・サービスの全文
- 関連するDBモデル定義
- 使用している定数・型定義

---

### Phase 2: 要件定義書の作成

**ファイル名:** `{ブランチ名のスラッシュをハイフンに変換}.requirement.md`  
**作成場所:** 改修対象リポジトリのルート

**必須セクション:**

- ブランチ名
- 背景・目的
- 対象リポジトリ / 参考リポジトリ（表形式）
- 改修内容（DB定義・バックエンド・フロントエンドの順）
- スコープ外（別PRで対応するもの）
- 参考ファイル一覧

**DBモデル名とテーブル名が一致しない場合は必ず注意セクションを追加する。**
調査でマッピングの不一致を発見したら、モデル名・テーブル名・カラム名のマッピング表を記載する。

---

### Phase 3: 設計書の作成

**ファイル名:** `{ブランチ名}.spec.md`  
**作成場所:** 改修対象リポジトリのルート

**必須セクション:**

- 変更ファイル一覧（表形式）
- 各ファイルの具体的な変更内容（実際のコードスニペット付き）
  - DBスキーマ変更（追加・変更のdiff形式 or 変更後コード）
  - マイグレーションSQL（自動生成部分の参考 + 手動追記部分）
  - 新規追加する関数の実装コード
  - 型定義の変更（before/after形式）
  - サービス層の変更（コードイメージ）
  - フロントエンドの変更（変更箇所と変更内容）
- テスト方針
- 実装順序（依存関係を考慮した順番）

---

### Phase 4: タスクリストの作成

**ファイル名:** `{ブランチ名}.task.md`  
**作成場所:** 改修対象リポジトリのルート

**形式:**

- Step 単位でグループ化（Step 1: スキーマ変更、Step 2: DB、Step 3: ロジック… のように）
- 各タスクはチェックボックス形式: `- [ ] **1-1** ...`
- 各タスクには変更するファイルパスと具体的な変更内容を記載
- コードスニペットが必要なものはインラインで記載
- 最後に動作確認タスクを入れる

**Stepの標準構成（プロジェクトに応じて調整）:**

1. スキーマ型定義変更（`schema/` 配下）
2. DBスキーマ変更 + マイグレーション生成 + データ移行SQL追記
3. バックエンド ヘルパー・共通ロジック追加
4. バックエンド サービス修正
5. フロントエンド修正
6. テスト更新・追加
7. 動作確認

---

## 出力サマリ

3ファイルの作成が完了したら、ユーザーに以下を伝える:

- 作成した3ファイルのパス
- 各ファイルの概要（1〜2行）
- 実装の注意点があれば補足（例: DBモデル名とテーブル名の不一致など）
