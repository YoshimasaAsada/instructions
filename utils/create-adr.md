# Create ADR

技術的な意思決定をADR（Architecture Decision Record）として構造化し記録する。

- **目的:** 技術選定・設計方針の背景・選択肢・決定理由を構造化して記録する
- **入力:** 意思決定の対象・背景・検討した選択肢
- **出力:** `ADR-{番号}.md`

---

## 事前確認事項

1. **意思決定の対象** — 何について決めたか（技術選定 / 設計方針 / 移行計画など）
2. **ADRの保存先** — `docs/adr/` など。プロジェクトの慣習に従う
3. **既存のADR** — 過去のADRがあればスタイルを合わせる

---

## 実行フロー

### Phase 1: 情報収集

ユーザーとの対話で以下を把握する:

- **背景**: なぜこの決定が必要になったか
- **選択肢**: 検討した代替案（最低2つ）
- **決定内容**: 何を選んだか
- **理由**: なぜその選択肢を選んだか
- **トレードオフ**: 選ばなかった案のメリットと、選んだ案のデメリット

不明な点はユーザーに質問して埋める。

---

### Phase 2: ADRの生成

以下のフォーマットで生成する:

```markdown
# ADR-{番号}: {タイトル}

## ステータス

{提案中 / 承認済み / 却下 / 廃止}

## 日付

{YYYY-MM-DD}

## 背景

（この決定が必要になった経緯・課題）

## 検討した選択肢

### 選択肢 A: {名前}
- **概要:** （簡潔な説明）
- **メリット:** （箇条書き）
- **デメリット:** （箇条書き）

### 選択肢 B: {名前}
- **概要:** （簡潔な説明）
- **メリット:** （箇条書き）
- **デメリット:** （箇条書き）

## 決定

{選択肢 X} を採用する。

## 理由

（なぜこの選択肢を選んだかの詳細な理由）

## 影響

- （この決定により変わること・必要になること）
- （受け入れるトレードオフ）

## 関連

- （関連するADR・Issue・ドキュメントへのリンク）
```

**ADR番号の採番:**
既存ADRがある場合は連番。なければ `001` から開始。

---

### Phase 3: 出力

- 生成したADRファイルのパス
- 内容のサマリー
- 関連する過去のADRがあれば参照を提示
