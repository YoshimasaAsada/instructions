# Type Review

型定義の品質を分析し、`any` / `unknown` / 不適切な型の使用を検出して改善案を提示する。

- **目的:** 型安全性を損なうパターンを検出し、適切な型への置き換えを提案する
- **入力:** 対象のファイル / ディレクトリ + 言語 + 厳格度（strict / moderate / loose）
- **出力:** 問題箇所の一覧（深刻度別）+ 改善後のコード

---

## 事前確認事項

1. **対象範囲** — ファイル / ディレクトリ / プロジェクト全体
2. **言語** — TypeScript / Python (type hints) / Go / その他
3. **厳格度** — 以下から選択
   - **strict**: `any` を全て排除、全ての関数に明示的な戻り値型
   - **moderate**: `any` を排除、推論可能な戻り値型は省略可
   - **loose**: 危険な `any` のみ修正（デフォルト）

---

## 実行フロー

### Phase 1: 型の問題を検出

対象コードを走査し、以下のパターンを検出する:

| 問題パターン | 深刻度 | 説明 |
|------------|--------|------|
| 明示的な `any` | 高 | 型安全性が完全に失われる |
| 暗黙の `any`（型推論失敗） | 高 | noImplicitAny で検出されるもの |
| `as` による型アサーション | 中 | ランタイムエラーの原因になりうる |
| `!`（non-null assertion） | 中 | null チェックの回避 |
| `unknown` の不適切な使用 | 中 | 適切なナローイングなしに使用 |
| union型の肥大化 | 低 | `string | number | boolean | ...` のような過剰な union |
| 型の重複定義 | 低 | 同一構造の型が複数箇所に存在 |

---

### Phase 2: 改善案の提示

検出した問題を深刻度順にまとめて提示する:

```
## 型の問題一覧

### 🔴 高（N件）
1. `path/to/file.ts:42` — `any` を `UserResponse` に変更
2. ...

### 🟡 中（N件）
3. `path/to/file.ts:78` — `as User` を型ガードに変更
4. ...

### 🔵 低（N件）
5. `path/to/types.ts:15` — `StatusType` と `ItemStatus` を統合
6. ...

修正する範囲を指定してください（全て / 高のみ / 選択）
```

---

### Phase 3: 修正の実施

ユーザーが承認した範囲の修正を実施する。

修正ルール:

- **既存のテストが通ること** — 型変更により既存テストが壊れないか確認
- **`any` の置き換え** — 適切な型が不明な場合は `unknown` + 型ガードを使う
- **型の新規作成が必要な場合** — プロジェクトの型定義ファイルの配置ルールに従う

---

### Phase 4: 完了報告

- 修正したファイル・箇所の一覧
- 修正前後の問題件数の比較
- type check の結果
