# Clean Commits

コミット履歴を整理しつつ、元の差分（最終的なコード変更）を完全に保証する。

- **目的:** コミット履歴をわかりやすく整理し、差分の同一性をハッシュで検証する
- **入力:** 整理対象のブランチとベースブランチ
- **出力:** 整理されたコミット履歴（force push コマンドは提示のみ・自動実行しない）

---

## 大原則

- **差分の保証が最優先。** 整理後の `git diff <ベースブランチ>...HEAD` が整理前と完全一致することを必ず確認する
- **push は自動実行しない。** コマンドを提示してユーザーに委ねる
- **作業前にバックアップブランチを作成する。** 何があっても元に戻せる状態を保つ

---

## 事前確認事項

以下が未確定の場合はユーザーに確認する:

1. **ベースブランチ** — どこからの差分を対象にするか（例: `main`, `develop`）
2. **整理方針** — 以下から選択（複数可）
   - **squash**: 複数コミットを1つにまとめる
   - **reword**: コミットメッセージを書き直す
   - **fixup**: wip・typo修正など細かいコミットを直前のコミットに吸収
   - **reorder**: コミットの順序を論理的に並べ直す
3. **まとめ方の粒度** — 1PRを1コミットにするか、機能単位に分けるかなど

---

## 実行フロー

### Phase 1: 事前準備

```bash
# 1. 現在のブランチ名を確認
git branch --show-current

# 2. バックアップブランチを作成（上書き禁止）
git branch backup/{ブランチ名}

# 3. 整理前の差分ハッシュを記録
git diff {ベースブランチ}...HEAD | shasum
# → この値を「元のハッシュ」として控えておく

# 4. 現在のコミット一覧を確認
git log --oneline {ベースブランチ}..HEAD
```

---

### Phase 2: 整理計画の立案

コミット一覧を読み、以下を決定してユーザーに提示・確認する:

**整理後のコミット構成案（例）:**

```
● feat: ユーザー認証機能を追加        ← squash: コミット①②③をまとめ
● fix: バリデーションエラーの修正      ← reword: メッセージを修正
● test: 認証機能のテストを追加        ← そのまま
```

**判断基準:**
- `wip`, `tmp`, `fix typo`, `fmt` などは直前の意味あるコミットに `fixup`
- 同じ機能・ファイル群への変更は `squash` でまとめる
- 無関係な変更が混在するコミットは分割を検討（ユーザーに確認してから実施）

---

### Phase 3: rebase の実行

```bash
# ベースブランチからの rebase を開始
git rebase -i {ベースブランチ}
```

rebase コマンドを直接実行するのではなく、**rebase-todo の内容を作成してユーザーに提示する。**

提示フォーマット:
```
以下の内容で rebase-todo を編集してください:

pick abc1234 feat: 認証機能の基盤を追加
squash def5678 wip: 途中まで
squash ghi9012 fix: レビュー指摘を修正
reword jkl3456 test: テスト追加
```

ユーザーが rebase を完了したら Phase 4 に進む。

---

### Phase 4: 差分の検証（必須）

```bash
# 整理後の差分ハッシュを取得
git diff {ベースブランチ}...HEAD | shasum
```

- **元のハッシュと一致 → Phase 5 へ**
- **不一致 → 即座にロールバック（下記）**

#### 不一致時のロールバック手順

```bash
# バックアップブランチの位置にリセット
git reset --hard backup/{ブランチ名}

# バックアップブランチを削除
git branch -d backup/{ブランチ名}
```

ロールバック後、不一致の原因（コンフリクト解消ミス・誤った squash など）をユーザーに報告する。

---

### Phase 5: 完了報告

以下をユーザーに伝える:

1. **整理後のコミット一覧** （`git log --oneline {ベースブランチ}..HEAD` の出力）
2. **差分検証結果** （ハッシュ一致を明記）
3. **バックアップブランチの案内** （不要なら削除コマンドを提示）
4. **force push コマンド** （実行はユーザーに委ねる）:
   ```bash
   # 差分確認済み。問題なければ以下を実行してください:
   git push --force-with-lease origin {ブランチ名}

   # バックアップが不要になったら:
   git branch -d backup/{ブランチ名}
   ```

> `--force` ではなく `--force-with-lease` を推奨する。
> リモートに自分以外のプッシュがあった場合に上書きを防ぐため。
