# Fix from Review Skill

`{ブランチ名}.review.md`（review.md で生成したレビューレポート）の指摘をもとに、
`create-docs.md` で作成した3つのドキュメントを修正するスキル。

**トリガー例:**

- 「レビュー指摘を反映して」
- 「review.md の内容をドキュメントに反映して」
- 「レビュー修正の設計をまとめて」

---

## 事前確認事項

以下が未確定の場合はユーザーに確認する:

1. **ブランチ名** — ドキュメントのファイル名プレフィックスになる
2. **ドキュメントの場所** — 改修対象リポジトリのルートを想定。異なる場合は確認

---

## 実行フロー

### Phase 1: ドキュメントの読み込み（並列実行）

以下の4ファイルを並列で読み込む:

| ファイル | 役割 |
|--------|------|
| `{ブランチ名}.review.md` | レビュー指摘の一覧（入力） |
| `{ブランチ名のスラッシュをハイフンに変換}.requirement.md` | 要件定義書（更新対象になりうる） |
| `{ブランチ名}.spec.md` | 設計書・コードスニペット（主な更新対象） |
| `{ブランチ名}.task.md` | タスクリスト（主な更新対象） |

---

### Phase 2: 指摘の分類

レビューレポートの「改善提案（Suggestions）」セクションを読み、各指摘を以下の3種別に分類する。

| 種別 | 定義 | 更新対象ドキュメント |
|------|------|-----------------|
| **A: 設計変更** | アーキテクチャ・責務・DB設計・インターフェース定義の変更を伴う | requirement.md / spec.md / task.md |
| **B: 実装修正** | 関数の切り出し・命名・ロジック・型定義など実装詳細の変更 | spec.md / task.md |
| **C: 軽微な修正** | コメント追加・フォーマット・変数名など影響範囲が局所的なもの | task.md のみ |

分類結果をユーザーに提示し、解釈が正しいか確認してから次のフェーズに進む。

---

### Phase 3: ドキュメントの更新

#### requirement.md（種別 A のみ更新）

- 変更が生じたセクション（改修内容・スコープ外など）を書き換える
- スコープ変更を伴う場合は「スコープ外」セクションに追記または移動する
- 変更箇所に `> ⚠️ レビュー指摘により変更 ({日付})` の引用ブロックを添える

#### spec.md（種別 A / B を更新）

- 各指摘に対応する変更ファイル・関数・コードスニペットを修正版に書き換える
- 修正前のコードを `before` ブロックで残し、修正後を `after` ブロックで示す形式を使う:

  ````
  **修正前:**
  ```ts
  // 指摘を受けたコード
  ```

  **修正後（レビュー反映）:**
  ```ts
  // 修正後のコード
  ```
  ````

- レビュー指摘番号（例: `Suggestion #1`）をコメントとして記載し、どの指摘に対応するか追跡可能にする

#### task.md（全種別を更新）

- 既存タスクが指摘対象の場合は、タスク内容をインラインで修正する
- 新規対応が必要な場合は「Step R: レビュー対応」セクションを末尾に追加し、以下の形式で列挙する:

  ```
  ## Step R: レビュー対応

  - [ ] **R-1** `path/to/file.ts` — {指摘内容の要約と具体的な修正内容}
  - [ ] **R-2** `path/to/other.ts` — {指摘内容の要約と具体的な修正内容}
  ```

- 種別 C（軽微な修正）はまとめて `**R-C** 軽微な修正（コメント・命名など）` として1行にまとめてよい

---

### Phase 4: 出力サマリ

更新が完了したら以下を出力する:

- 更新した3ファイルのパス
- 分類別の指摘件数（A: N件 / B: N件 / C: N件）
- 設計に影響があった指摘の要約（特に requirement.md を変更した場合）
- 実装時の注意点（指摘対応で生じた依存関係の変化など）
