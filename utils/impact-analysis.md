# Impact Analysis

変更対象のファイル・関数から、影響を受ける箇所を洗い出す。

- **目的:** 変更による直接・間接の影響範囲を特定し、必要な対応を提案する
- **入力:** 変更対象（ファイルパス / 関数名 / テーブル名）+ 変更の種類
- **出力:** 影響マップ（直接影響・間接影響・対応方針）

---

## 事前確認事項

1. **変更対象** — ファイルパス / 関数名 / クラス名 / DBテーブル名
2. **変更の種類** — シグネチャ変更 / 削除 / リネーム / ロジック変更 / スキーマ変更

---

## 実行フロー

### Phase 1: 直接依存の調査

変更対象を参照している箇所を検索する:

- **import / require** — 対象をインポートしているファイル
- **関数呼び出し** — 対象関数を呼んでいる箇所
- **型参照** — 対象の型を使っているファイル
- **DB参照** — テーブル変更の場合、クエリ・ORM・マイグレーションへの影響

---

### Phase 2: 間接依存の調査

直接依存しているファイル・関数が、さらに他から参照されていないかを再帰的に追跡する。

追跡の深さはデフォルト2階層。深い追跡が必要な場合はユーザーに確認する。

---

### Phase 3: 影響マップの作成

調査結果を以下の形式で提示する:

```
## 影響マップ

### 変更対象
`path/to/target.ts` — `functionName`

### 直接影響（N件）
| ファイル | 箇所 | 影響内容 |
|---------|------|---------|
| `path/to/caller.ts` | `callerFunction():42` | 引数の型が変わる |
| `path/to/test.ts` | `describe("target"):15` | テストの修正が必要 |
| ... | ... | ... |

### 間接影響（N件）
| ファイル | 箇所 | 経路 | 影響内容 |
|---------|------|------|---------|
| `path/to/handler.ts` | `handleRequest():28` | caller → handler | 戻り値の型が変わる |
| ... | ... | ... | ... |

### 影響なし（確認済み）
- `path/to/unrelated.ts` — import しているが該当関数は未使用
```

---

### Phase 4: 対応方針の提案

影響箇所ごとに、必要な対応を提案する:

- **修正が必要** — 具体的な修正内容
- **テスト修正が必要** — どのテストケースに影響するか
- **確認のみ** — 動作に影響はないが念のため確認すべき箇所
- **対応不要** — 影響がない理由
