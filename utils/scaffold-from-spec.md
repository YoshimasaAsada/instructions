# Scaffold from Spec

設計書（spec.md）を入力に、スキャフォールドコードを生成する。

- **目的:** 設計書のコードスニペットと実装順序に基づき、動作する雛形コードを生成する
- **入力:** `{ブランチ名}.spec.md` / `{ブランチ名}.task.md`（任意）
- **出力:** 設計に対応するソースコード・テスト雛形

---

## 事前確認事項

1. **設計書のパス** — `{ブランチ名}.spec.md` の場所
2. **タスクリストのパス** — `{ブランチ名}.task.md`（あれば）
3. **生成範囲** — 全体を一括生成するか、Step 単位で段階的に進めるか
4. **テストも生成するか** — テストファイルの雛形も一緒に作るかどうか

---

## 実行フロー

### Phase 1: 設計書の読み込み

以下を読み込む:

- `{ブランチ名}.spec.md` — 変更ファイル一覧・コードスニペット
- `{ブランチ名}.task.md` — 実装順序と依存関係（あれば）
- 改修対象リポジトリの既存コード — 設計書で言及されているファイル

---

### Phase 2: 生成計画の提示

設計書の「実装順序」または task.md の Step に基づき、生成するファイルの一覧をユーザーに提示する。

**提示フォーマット:**

```
以下の順序でコードを生成します:

Step 1: スキーマ変更
  - path/to/schema.prisma（モデル追加）
  - path/to/types.ts（型定義追加）

Step 2: バックエンド
  - path/to/service.ts（関数追加: createFoo, updateFoo）
  - path/to/repository.ts（新規作成）

生成してよいですか？
```

---

### Phase 3: コード生成

Step 単位で順番に生成する（依存関係があるため並列にしない）。

各ファイルの生成ルール:

- **既存ファイルの変更**: 設計書の before/after を参照し、差分だけを適用する
- **新規ファイルの作成**: 設計書のコードスニペットをベースに、周囲のコードスタイル（命名規則・import形式・ディレクトリ構成）に合わせる
- **テスト**: 対応するテストファイルが指定されていれば同時に生成する

Step ごとに以下を出力する:
- 生成・変更したファイルのパス
- 変更の概要
- task.md がある場合、対応するタスクのチェックを入れる

---

### Phase 4: 検証

生成完了後に以下を実施:

```bash
# 構文チェック（プロジェクトの lint / type check コマンドがあれば実行）
npm run lint   # or yarn lint, etc.
npm run typecheck  # or tsc --noEmit, etc.
```

エラーがあれば修正し、エラーがなければ完了報告へ。

---

### Phase 5: 完了報告

- 生成・変更したファイルの一覧
- task.md の完了状況
- lint / type check の結果
- 手動で確認すべきポイント（例: 環境変数の追加、マイグレーションの実行など）
