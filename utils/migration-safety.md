# Migration Safety Check

DBスキーマ変更の影響範囲を分析し、安全な実行手順とロールバック計画を生成する。

- **目的:** マイグレーションのリスク評価・影響範囲特定・ロールバックSQL生成を行う
- **入力:** マイグレーションSQL / ORMスキーマ差分 + 対象環境 + テーブル規模
- **出力:** リスク評価 + 実行計画 + ロールバック手順・SQL

---

## 事前確認事項

1. **マイグレーション内容** — SQLファイル / ORMスキーマ変更の差分
2. **対象環境** — 本番 / ステージング / 開発
3. **テーブルの規模感** — 大規模テーブル（100万行以上）かどうか
4. **ダウンタイム許容** — 無停止が必須か、メンテナンスウィンドウがあるか

---

## 実行フロー

### Phase 1: 変更内容の分析

マイグレーションの内容を読み込み、以下に分類する:

| 操作 | リスクレベル | 例 |
|------|-----------|---|
| カラム追加（NULL許容） | 低 | `ALTER TABLE ADD COLUMN ... NULL` |
| インデックス追加 | 中 | 大規模テーブルではロックの可能性 |
| カラム追加（NOT NULL） | 中 | デフォルト値の設定が必要 |
| カラム型変更 | 高 | データ損失の可能性 |
| カラム削除 | 高 | コードが参照していないか要確認 |
| テーブル削除 | 最高 | 復旧不可能 |
| カラムリネーム | 高 | アプリケーションコードの同時変更が必要 |

---

### Phase 2: 影響範囲の特定

変更対象のテーブル・カラムを参照しているコードを洗い出す:

- **ORMモデル / エンティティ** — 変更対象テーブルに対応するモデル定義
- **クエリ / リポジトリ** — 変更対象カラムを参照するクエリ
- **API / レスポンス** — 変更がAPIレスポンスに影響するか
- **バッチ処理** — 定期実行ジョブで参照していないか
- **他サービス** — マイクロサービス構成の場合、他サービスが直接参照していないか

---

### Phase 3: 安全な実行計画の提示

```
## マイグレーション計画

### 変更の概要
（変更内容のサマリー）

### リスク評価: {低 / 中 / 高 / 最高}
（リスクの理由）

### 事前作業
1. （バックアップの取得方法）
2. （必要なコード変更のデプロイ順序）

### 実行手順
1. （マイグレーションの実行コマンド）
2. （実行後の確認項目）

### ロールバック手順
1. （マイグレーションを元に戻すSQL）
2. （コードのロールバック手順）

### 注意事項
- （ロック時間の見積もり）
- （デプロイ順序の制約）
- （データ移行が必要な場合の手順）
```

---

### Phase 4: ロールバックSQLの生成

マイグレーションに対応する逆操作のSQLを生成する:

- `ADD COLUMN` → `DROP COLUMN`
- `CREATE INDEX` → `DROP INDEX`
- `ALTER COLUMN TYPE` → 元の型に戻すSQL + データ変換

ロールバックSQLが安全に実行できない操作（データ損失を伴う型変更など）は、その旨を明記する。

---

### Phase 5: 完了報告

- リスク評価の結果
- 影響を受けるファイル一覧
- 推奨するデプロイ順序
- ロールバック手順とSQL
- 手動確認が必要な項目
