# Generate Tests

対象のコードを読み取り、テストケースを網羅的に洗い出してテストコードを生成する。

- **目的:** 正常系・境界値・異常系を網羅したテストケースを設計し、テストコードを生成する
- **入力:** テスト対象のファイルパス / 関数名 / クラス名
- **出力:** テストケース一覧 + テストコード

---

## 事前確認事項

1. **対象** — ファイルパス / 関数名 / クラス名
2. **テストフレームワーク** — プロジェクトで使用しているもの（Jest, Vitest, pytest, Go testing など）。不明ならプロジェクトの設定ファイルから自動検出する
3. **テスト方針** — 以下から選択
   - ユニットテスト（関数単位、依存はモック）
   - インテグレーションテスト（実際のDB・外部サービスを使用）
   - 両方

---

## 実行フロー

### Phase 1: 対象コードの分析

対象のコードを読み込み、以下を把握する:

- 関数のシグネチャ（引数・戻り値の型）
- 分岐条件（if / switch / 三項演算子 / 早期リターン）
- 外部依存（DB呼び出し・API呼び出し・ファイル操作など）
- エラーハンドリング（throw / Error / 例外）
- 既存テストの有無（あれば読み込んでスタイルを合わせる）

---

### Phase 2: テストケースの洗い出し

以下のカテゴリでテストケースを列挙し、ユーザーに提示する:

| カテゴリ | 説明 |
|---------|------|
| 正常系 | 標準的な入力で期待する出力が得られること |
| 境界値 | 0, 空文字, 空配列, null, 最大値など |
| 異常系 | 不正な入力・存在しないリソース・権限不足など |
| エッジケース | 並行処理・タイムアウト・大量データなど |

**提示フォーマット:**

```
テストケース一覧:

✅ 正常系
  1. {入力の説明} → {期待する結果}
  2. ...

⚠️ 境界値
  3. {入力の説明} → {期待する結果}
  4. ...

❌ 異常系
  5. {入力の説明} → {期待するエラー}
  6. ...

追加・削除したいケースはありますか？
```

---

### Phase 3: テストコードの生成

ユーザーの確認後、テストコードを生成する。

生成ルール:

- **既存テストのスタイルに合わせる** — describe/it の粒度、モックの書き方、アサーションライブラリなど
- **テストごとに独立** — テスト間で状態を共有しない
- **Arrange-Act-Assert パターン** — セットアップ・実行・検証を明確に分離
- **テスト名は日本語 or 英語** — 既存テストに合わせる。なければプロジェクトの主言語に従う
- **モック** — ユニットテストの場合、外部依存はモック化。インテグレーションテストでは実サービスを使用

---

### Phase 4: 実行と確認

```bash
# テスト実行
npm run test -- {テストファイルパス}
# or
pytest {テストファイルパス}
```

- **全件パス** → 完了報告へ
- **失敗あり** → 原因を分析して修正し、再実行

---

### Phase 5: 完了報告

- 生成したテストファイルのパス
- テストケース数（正常系 N件 / 境界値 N件 / 異常系 N件）
- テスト実行結果
- カバレッジ（取得可能であれば）
